# ikramrana.com

Website source and production export for [ikramrana.com](https://ikramrana.com).

## Repository layout

- `client/src/`: editable React and TypeScript application source
- `client/public/`: public source assets copied into frontend builds
- `server/index.ts`: Express production server for the Node deployment option
- `shared/`: code shared by the client and server
- `ops-kit/`: PHP-protected digital-product dashboard
- `assets/`, `index.html`, `robots.txt`, and `sitemap.xml` at the repository root: current static production export

Do not edit minified files under the root `assets/` directory. Make application changes under `client/src/`, then create a new build.

## Requirements

- Node.js 22
- pnpm, managed through Corepack

Enable pnpm once if it is not already available:

```bash
corepack enable
```

You can also prefix pnpm commands with `corepack`, for example `corepack pnpm install`.

## Initial setup

```bash
git clone https://github.com/barranainc/ikramrana.com.git
cd ikramrana.com
corepack pnpm install --frozen-lockfile
```

## Development

```bash
corepack pnpm dev
```

Vite starts on port 3000 or the next available port.

Before committing a change, run:

```bash
corepack pnpm check
corepack pnpm build
```

The build command produces:

- `dist/public/`: compiled frontend
- `dist/index.js`: bundled Express server

The `dist/` directory is intentionally ignored by Git.

## Git workflow

Create changes on a short-lived branch rather than committing directly to `main`:

```bash
git checkout main
git pull
git checkout -b feature/descriptive-name
```

After editing and validating:

```bash
git add .
git commit -m "Describe the change"
git push -u origin feature/descriptive-name
```

Open a pull request, obtain review when appropriate, and merge it into `main`. Commit and push every completed change so other developers receive the same source.

## Deployment

The current root export is configured for Hostinger/LiteSpeed through `.htaccess`. It serves existing files directly and routes all other URLs to the single-page application.

The repository also supports a Node deployment using:

```bash
corepack pnpm build
corepack pnpm start
```

The production host currently appears to use the static Hostinger/LiteSpeed approach. The exact upload or CI/CD procedure is not stored in this repository and must be confirmed with the hosting account owner before deploying.

For a static deployment, publish the contents of `dist/public/` to the web root while preserving server-only files and directories that are not generated by Vite, particularly:

- `.htaccess`
- `ops-kit/`
- protected downloadable assets used by `ops-kit/`

Do not blindly delete or replace the entire web root. Back it up first and verify `/`, a client-side route, `/robots.txt`, `/sitemap.xml`, and `/ops-kit/` after deployment.

For the current approved release, use `WEBSITE_CHANGES_AND_LIVE_DEPLOYMENT_GUIDE.md`. It contains the complete change register, production archive checksum, Hostinger procedure, verification checklist, and rollback instructions.

## Environment variables and secrets

Environment files are ignored by Git. Never commit passwords, API keys, access tokens, cookie-signing keys, or production `.env` files.

The current `ops-kit/index.php` contains an embedded access password and cookie-signing secret inherited from the production export. These credentials must be rotated and moved to server-side environment variables before treating the access gate as secure.

Document required variable names in a committed `.env.example`, but use placeholder values only.

## Important maintenance notes

- Update content and UI in `client/src/`, not in the compiled root bundle.
- Keep `pnpm-lock.yaml` committed and use `--frozen-lockfile` in CI and deployments.
- Do not commit `node_modules/`, `dist/`, logs, or local environment files.
- Check both desktop and mobile layouts after visual changes.
- Preserve analytics, SEO metadata, redirects, the sitemap, and the SPA fallback when changing deployment configuration.
- Treat PDFs, ZIP files, and other downloadable products as potentially sensitive and review whether they should live in Git.
